# 0004 Public Interface Index Evidence Kind Proposal

Status: Implemented in SpecPM 0.2.0.

## Proposal

SpecPM recognizes `public_interface_index` as a known `BoundarySpec`
`evidence.kind`.

The evidence kind represents a deterministic, local-only summary of a package's
public interface surface. It is intended for generated or tool-assisted
packages that need to cite compact API evidence without embedding raw source
code in the `BoundarySpec`.

Example:

```yaml
evidence:
  - id: public_interface_index
    kind: public_interface_index
    path: public-interface-index.json
    supports:
      - interfaces.inbound
      - interfaces.inbound.package.flask
```

The referenced artifact is ordinary package data. SpecPM rejects paths that
escape the package root and reports missing evidence paths as validator
warnings, consistent with the existing evidence path behavior. SpecPM does not
execute the artifact, import package code, install dependencies, run build
tools, or interpret package-provided instructions.

## Context

SpecHarvester now emits `public-interface-index.json` artifacts for static
analyzers such as Python `ast` and JavaScript/TypeScript manifest/export
analysis. Those artifacts summarize packages, entrypoints, exported symbols,
source digests, analyzer provenance, diagnostics, and summary status.

During smoke validation of generated candidates, SpecPM currently reports:

```text
unknown_evidence_kind: Unknown evidence kind: public_interface_index
```

This warning was correct under RFC 0001 because the MVP evidence kind
vocabulary was:

```text
documentation
test
source
example
schema
foreign_spec
package_manifest
adr
commit
manual_assertion
unknown
```

However, the existing vocabulary does not distinguish compact analyzer output
from raw source evidence or schema evidence. Treating the artifact as `source`
would overstate what is being cited. Treating it as `schema` would imply a data
contract rather than observed public API evidence. Treating it as `unknown`
would preserve validation silence only by losing useful semantics.

## Intended Meaning

`public_interface_index` means:

- the artifact summarizes public API or package interface evidence;
- the artifact was generated by a deterministic local analyzer or equivalent
  static process;
- the artifact is evidence for interface declarations and related capability
  claims;
- the artifact is not itself an implementation binding;
- the artifact is not accepted registry truth without package review;
- the artifact is not executable instructions.

The evidence kind should be usable by generated packages and manually authored
packages. Manual authors may use it when they vendor or publish a stable
interface index generated by their own tooling.

## Minimal Artifact Expectations

SpecPM does not need to fully parse the artifact in the first implementation.
The first validator change is intentionally narrow:

1. Add `public_interface_index` to the known evidence kind vocabulary.
2. Continue validating `path` using the existing package-root path rules.
3. Continue warning when the referenced file does not exist.
4. Do not execute, import, compile, or network-fetch anything.

A later profile may validate the artifact shape. The current SpecHarvester
shape includes fields such as:

```text
kind: PublicInterfaceIndex
schemaVersion: integer
sourceRevision: string
analyzers: list
packages: list
diagnostics: list
summary: object
```

Shape validation is useful, but should be separate from the initial vocabulary
addition so that SpecPM can accept the evidence kind without coupling itself to
one producer too early.

## Relationship to Existing Evidence Kinds

`public_interface_index` is distinct from:

- `source`: raw or direct implementation files;
- `schema`: formal schema contracts such as JSON Schema, OpenAPI, or similar;
- `package_manifest`: package-manager metadata such as `pyproject.toml`,
  `package.json`, `go.mod`, or `Package.swift`;
- `documentation`: human-facing explanatory material;
- `foreign_spec`: another specification system or upstream spec artifact.

It is closest to an analyzer-derived evidence summary. Its purpose is to let a
package cite the public interface surface without requiring consumers or weak
models to read raw source trees.

## Trust Boundary

SpecPM must treat `public_interface_index` as untrusted package content.

SpecPM may:

- check that the path is inside the package root;
- check that the path exists;
- expose the evidence item through inspect, registry, and JSON report surfaces;
- later validate a documented JSON shape if the profile is accepted.

SpecPM must not:

- execute analyzers while validating a package;
- trust analyzer output as registry acceptance;
- treat symbol names as proof of runtime behavior;
- follow network links from the artifact;
- run package code, package scripts, tests, or build systems;
- let the artifact override host policy or validation behavior.

## Validator Impact

The minimal implementation is additive and backward-compatible:

```python
KNOWN_EVIDENCE_KINDS = {
    ...
    "public_interface_index",
}
```

Existing packages remain valid. Packages that already emit
`kind: public_interface_index` stop producing the `unknown_evidence_kind`
warning, while all existing path and support-target checks continue to apply.

This proposal does not require a new package `apiVersion`. It is an additive
vocabulary expansion for `specpm.dev/v0.1`.

## Registry and JSON Contract Impact

Registry and `inspect --json` payloads already expose evidence entries as data.
Adding a known evidence kind does not require a new registry endpoint or payload
shape.

Consumers that switch on `evidence.kind` should treat unknown values
conservatively. After this proposal is implemented, they may display
`public_interface_index` with a dedicated label such as "Public Interface
Index" or "Analyzer Interface Summary".

## SpecHarvester Compatibility

This proposal exists because SpecHarvester already produces candidates with:

```yaml
evidence:
  - id: public_interface_index
    kind: public_interface_index
    path: public-interface-index.json
```

The generated artifact is designed to support weak-model drafting and review by
providing compact deterministic material instead of raw source dumps.

SpecHarvester still needs a separate cleanup for generated `supports` targets
that SpecPM does not currently declare, such as
`provides.capabilities.intentIds`. That issue is related to support-target
vocabulary, not to evidence-kind vocabulary, and should be handled separately.

## Non-Goals

This proposal does not:

- require SpecPM to run analyzers;
- require SpecPM to parse every interface-index profile immediately;
- define a universal AST or symbol graph schema;
- make analyzer output accepted registry truth;
- change `SpecPackage` or `BoundarySpec` top-level fields;
- change package archive format;
- change public registry endpoint families;
- solve generated evidence support-target warnings.

## Acceptance Criteria

The implementation is complete when:

- maintainers agree that `public_interface_index` is a stable evidence kind;
- validator tests cover the evidence kind as known;
- path validation behavior remains unchanged;
- `unknown_evidence_kind` is no longer emitted for
  `kind: public_interface_index`;
- documentation mentions that the artifact is untrusted data and must not be
  executed.

## Open Questions

1. Should SpecPM later validate a minimal JSON profile for
   `public-interface-index.json`, or should shape validation remain owned by the
   producer?
2. Should `public_interface_index` be allowed to support only `interfaces.*`
   targets, or also capability and intent summary claims?
3. Should the RFC 0001 text be amended directly, or should this remain a
   post-RFC decision document until the next package schema revision?
