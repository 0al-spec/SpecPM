# Curated Accepted Artifact Lifecycle

Status: Draft
Updated: 2026-06-10
Scope: SpecPM public index lifecycle for maintainer-curated accepted artifacts
created from producer-generated candidates

## Purpose

This document defines the lifecycle boundary between producer-generated
candidate bundles and maintainer-curated accepted artifacts.

The short rule is:

```text
generated candidate -> review evidence
curated accepted artifact -> registry source
```

For example, `public-index/generated/xyflow.*` records what SpecHarvester
emitted. `public-index/curated/xyflow.*` records what SpecPM maintainers
accepted after review.

## 1. Generated Candidate Immutability

Generated candidates are reproducible producer evidence. Maintainers should not
edit `public-index/generated/<package_id>/<version>/specpm.yaml`,
`specs/*.spec.yaml`, `producer-receipt.json`, validation reports, diagnostics
reports, or harvested evidence in place to polish accepted registry metadata.

If a generated candidate is wrong, maintainers should either request
regeneration or create a separate curated artifact that cites the generated
candidate as evidence. The producer receipt must continue to describe the
bytes the producer emitted, not later maintainer edits.

## 2. Curated Artifact Ownership

Curated accepted artifacts are maintainer-owned registry sources. They own the
human-reviewed public metadata, including:

- `metadata.summary`;
- accepted capabilities;
- accepted intent wording and capability `intentIds`;
- evidence wording and claim support;
- package boundary wording;
- removal of `preview_only` when maintainers accept non-preview registry
  metadata.

Curated artifacts must not imply that maintainer-authored summaries,
capabilities, constraints, or evidence wording were generated by the producer.

## 3. New Harvest Update Flow

A new SpecHarvester run is review input, not an automatic upgrade of accepted
registry metadata. The update flow is:

1. Generate a new candidate bundle under `public-index/generated` or attach it
   as producer evidence.
2. Run producer-bundle, package-set, AI-enrichment, and materialization
   preflights when applicable.
3. Compare the new generated candidate against the existing curated artifact.
4. Review changed claims, source revisions, evidence, capabilities, versions,
   and relations.
5. Open a targeted curated artifact update pull request.

Maintainers may accept some generated changes, rewrite others, or reject the
new candidate entirely. The curated artifact changes only through ordinary
review.

## 4. Evidence Chain Through `foreignArtifacts`

A curated artifact should preserve a reviewable evidence chain through
`foreignArtifacts` or equivalent provenance fields. When available, cite:

- the generated candidate `specpm.yaml`;
- `producer-receipt.json`;
- `validation-report.json`;
- `diagnostics.json`;
- package-set handoff reports;
- materialization reports or accepted-source review notes;
- AI enrichment proposals, when they were used as review evidence.

These links are evidence. They do not transfer registry authority to the
producer, model provider, preflight command, or generated candidate.

## 5. `preview_only` Removal Rule

Generated candidates should remain `preview_only` when they are producer drafts
or otherwise unreviewed. Removing `preview_only` is a maintainer acceptance
act.

A curated accepted artifact may omit `preview_only` only after maintainers
review the package claims, source evidence, version, package boundary, and any
selected relations. Passing validation, producer-bundle preflight,
AI-enrichment preflight, or materialization dry run is evidence for review, not
approval to publish non-preview registry metadata.

## 6. Relation Acceptance Boundary

Package relations are accepted separately from package candidates.

Producer handoff relation proposals are evidence. Registry-visible relations
exist only when maintainers add explicit entries to
`public-index/accepted-packages.yml` `relations[]`. Accepting a package does not
automatically accept its proposed relations, and accepting a relation does not
grant trust, lifecycle, or capability inheritance between packages.

For a package-set such as `xyflow.workspace`, maintainers must explicitly
select both member packages and relation IDs before `/v0` can publish
`contains` relation metadata.

## 7. Refresh Decision Records

A fresh producer run can be valuable even when it does not require a registry
update. Maintainers should record that outcome as a
`SpecPMGeneratedCandidateRefreshDecision` with `status: no_update_required`,
`updateNeeded: false`, and `reason: no_contract_delta` when the generated
contract bytes, accepted curated claims, source revision, package relations, and
evidence support have no meaningful accepted contract delta.

This avoids treating producer receipt churn, local run paths, newly emitted
advisory quality reports, or equivalent generated candidates as reasons to
churn accepted registry metadata.

## Non-Goals

This lifecycle does not add automatic package acceptance, producer-owned
registry publication, signature trust enforcement, semantic package selection,
dependency solving, relation inheritance, or request-time registry mutation.
