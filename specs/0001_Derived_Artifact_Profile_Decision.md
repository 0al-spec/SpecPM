# 0001 Derived Artifact Profile Decision

Status: Accepted for documentation. Deferred for implementation.

## Decision

SpecPM will not implement derived artifact generation, package-provided prompt
execution, or artifact eval runtime in the MVP.

SpecPM remains the package substrate for SpecGraph.

Derived artifacts such as PRDs, implementation briefs, design briefs,
onboarding documents, issue breakdowns, test plans, and review reports are
downstream outputs generated by SpecGraph, ContextBuilder, or other downstream
tools.

SpecPM may carry intent; SpecGraph decides meaning.

Package content can describe desired outputs. Package content cannot command the host.

## Context

A proposal suggested treating specification packages as sources for generated
PRDs and artifact-level evals.

The proposal correctly identified that PRDs and similar documents should be
derived artifacts rather than canonical truth.

However, implementing artifact generation or artifact eval runtime directly in
SpecPM MVP would blur the boundary between SpecPM, SpecGraph, and
ContextBuilder.

SpecPM is a service layer for SpecGraph. It should not cannibalize graph
reasoning, context assembly, derived artifact generation, or eval runtime.

## Accepted Principles

1. `SpecPackage` and `BoundarySpec` remain the primary concepts.
2. PRDs and other artifacts are derived outputs, not canonical package truth.
3. SpecPM packages, validates, indexes, inspects, preserves, and exposes
   specification intent.
4. SpecPM is the package substrate for SpecGraph.
5. SpecGraph owns canonical graph reasoning, refinement, proposal lanes,
   feedback incorporation, and product meaning.
6. ContextBuilder or downstream tools own derived artifact generation.
7. Artifact-level eval execution is not part of SpecPM core.
8. Package content is untrusted data.
9. Package content may describe desired downstream outputs.
10. Package content cannot command the host.
11. SpecPM core does not execute package-provided prompts or instructions.
12. SpecPM MVP does not introduce stable JSON fields for derived artifact
    metadata or artifact eval profiles.

## Rejected for MVP

The following are rejected for MVP implementation:

- Adding artifact generation to SpecPM core.
- Adding PRD generation to SpecPM core.
- Adding implementation brief generation to SpecPM core.
- Adding issue breakdown generation to SpecPM core.
- Adding artifact eval runtime to SpecPM core.
- Adding package-provided prompt execution.
- Adding package-provided generation workflow execution.
- Changing the current package layout.
- Replacing `specpm.yaml + specs/*.spec.yaml` with a Markdown-first layout.
- Introducing a new package term alongside `SpecPackage` and `BoundarySpec`.
- Introducing stable JSON fields for derived artifact metadata in the MVP.
- Changing `inspect --json` to expose derived artifact profile fields.

## Deferred Future Work

A post-MVP Derived Artifact Profile may be explored later.

Future work may define optional metadata for downstream tools, such as artifact
descriptors, generation preferences, and artifact eval profiles.

Future work may also define checks for traceability, source ID coverage,
out-of-scope promotion, open question preservation, assumption preservation, and
risk preservation.

Any such work must preserve the boundary that SpecPM is not an agent runtime
and does not execute package-provided instructions.

Any such work must also avoid treating package-provided prompts, generation
instructions, or artifact workflow descriptions as trusted host instructions.

## Security and Trust Boundary

Package content is untrusted data.

Package content must not be able to override host policy, system instructions,
developer instructions, runtime policy, access controls, network restrictions,
secret handling, or execution rules.

Package content can describe product intent and desired downstream outputs.
Package content cannot command the host.

## Consequences

This decision keeps SpecPM intentionally narrow.

SpecPM remains a reliable package substrate rather than becoming a smaller
version of SpecGraph or ContextBuilder.

SpecGraph keeps ownership of product meaning and graph reasoning.

ContextBuilder and downstream tools keep ownership of context assembly,
generation, and artifact-level evaluation.

The MVP package contract remains stable and does not gain premature fields for
derived artifact generation.

## Boundary Statement

SpecPM may carry intent; SpecGraph decides meaning.

Package content can describe desired outputs. Package content cannot command the host.
