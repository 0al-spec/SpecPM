# 0001 Derived Artifact Profile Decision

Status: Accepted for documentation. Deferred for implementation.

## Decision

SpecPM will not implement derived artifact generation, package-provided prompt
execution, or artifact eval runtime in the MVP.

SpecPM remains the package substrate for SpecGraph. It packages, validates,
indexes, inspects, diffs, and preserves reusable specification intent.

Derived artifacts such as PRDs, implementation briefs, design briefs,
onboarding documents, issue breakdowns, and test plans are downstream outputs
generated by SpecGraph, ContextBuilder, or other downstream tools.

## Context

A proposal suggested treating specification packages as sources for generated
PRDs and artifact-level evals. The proposal correctly identified that PRDs
should be derived artifacts rather than canonical truth.

However, implementing this directly in SpecPM MVP would blur the boundary
between SpecPM, SpecGraph, and ContextBuilder.

SpecPM is a service layer for SpecGraph. It should not own graph reasoning,
context assembly, artifact generation, package-provided prompt execution, or
artifact eval runtime.

## Accepted Principles

1. `SpecPackage` and `BoundarySpec` remain the primary concepts.
2. PRDs and other artifacts are derived outputs, not canonical package truth.
3. SpecPM packages, validates, indexes, inspects, diffs, and preserves
   specification intent.
4. SpecGraph owns canonical graph reasoning, refinement, and proposal lanes.
5. ContextBuilder or downstream tools own derived artifact generation.
6. Package content is untrusted data.
7. Package content may describe desired outputs.
8. Package content cannot command the host.
9. SpecPM core does not execute package-provided prompts or instructions.
10. SpecPM MVP does not introduce stable JSON fields for derived artifact
    metadata.

## Rejected for MVP

The following are rejected for MVP implementation:

- Adding artifact generation to SpecPM core.
- Adding PRD generation to SpecPM core.
- Adding artifact eval runtime to SpecPM core.
- Adding package-provided prompt execution.
- Replacing the current package layout with a Markdown-first layout.
- Introducing a new MSP term alongside `SpecPackage` and `BoundarySpec`.
- Introducing stable JSON fields for artifact metadata in the MVP.

## Deferred Future Work

A post-MVP Derived Artifact Profile may be explored later.

Future work may define optional metadata for downstream tools, such as artifact
descriptors, generation preferences, and artifact eval profiles.

Any such work must preserve the boundary that SpecPM is not an agent runtime
and does not execute package-provided instructions.

## Boundary Statement

SpecPM may carry intent; SpecGraph decides meaning.

Package content can describe desired outputs. Package content cannot command the
host.
