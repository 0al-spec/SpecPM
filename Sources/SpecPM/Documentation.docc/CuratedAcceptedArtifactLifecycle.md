# Curated Accepted Artifact Lifecycle

SpecPM separates producer-generated candidates from maintainer-curated accepted
artifacts.

```text
generated candidate -> review evidence
curated accepted artifact -> registry source
```

`public-index/generated/xyflow.*` records what SpecHarvester emitted.
`public-index/curated/xyflow.*` records what SpecPM maintainers accepted after
review.

## Generated Candidate Immutability

Generated candidates are reproducible producer evidence. Maintainers should not
edit generated `specpm.yaml`, `specs/*.spec.yaml`, `producer-receipt.json`,
validation reports, diagnostics reports, or harvested evidence in place to
polish accepted registry metadata.

If a generated candidate is wrong, request regeneration or create a separate
curated artifact that cites the generated candidate as evidence. The producer
receipt must keep describing the bytes the producer emitted.

## Curated Artifact Ownership

Curated accepted artifacts are maintainer-owned registry sources. They own
human-reviewed summaries, accepted capabilities, accepted intent wording,
evidence wording, package boundary wording, and `preview_only` removal.

Curated artifacts must not imply that maintainer-authored text was generated by
the producer.

## New Harvest Update Flow

A new SpecHarvester run is review input, not an automatic upgrade. Maintainers
should generate or attach the new candidate, run relevant preflights, compare
against the existing curated artifact, review changed claims and evidence, and
open a targeted curated update pull request.

Maintainers may accept some generated changes, rewrite others, or reject the
new candidate entirely.

## Evidence Chain

Curated artifacts should cite producer evidence through `foreignArtifacts` or
equivalent provenance fields. When available, cite the generated candidate,
`producer-receipt.json`, `validation-report.json`, `diagnostics.json`,
package-set handoff reports, materialization reports, accepted-source review
notes, and AI enrichment proposals used during review.

These links are evidence. They do not transfer registry authority to the
producer, model provider, preflight command, or generated candidate.

## Preview Boundary

Generated candidates should remain `preview_only` when they are producer drafts
or otherwise unreviewed. Removing `preview_only` is a maintainer acceptance act.

A curated accepted artifact may omit `preview_only` only after maintainers
review package claims, source evidence, version, package boundary, and selected
relations. Passing validation, producer-bundle preflight, AI-enrichment
preflight, or materialization dry run is evidence for review, not approval to
publish non-preview registry metadata.

## Relation Acceptance

Package relations are accepted separately from package candidates. Producer
handoff relation proposals are evidence. Registry-visible relations exist only
when maintainers add explicit entries to
`public-index/accepted-packages.yml` `relations[]`.

Accepting a package does not automatically accept its proposed relations, and
accepting a relation does not grant trust, lifecycle, or capability inheritance.
